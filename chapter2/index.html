    Chapter 2 · Research & Universal Usability

    Design decisions need evidence.

    A website becomes stronger when planning is tested against real users, real goals, and real situations.

    Opening question How do we know if a website is easy to use?

    Today’s path

    1. Question assumptions about users
    2. Compare research methods
    3. Build personas and scenarios
    4. Sketch, prototype, and test early

    Use the arrow keys or the Next button to move through the deck.

    Main idea

    Best practices are not enough.

    Good web design uses familiar patterns, but the strongest decisions come from learning from users throughout the design process.

    Assumption

    “Users will understand this because it makes sense to us.”

    Evidence

    “Three test users tried the task. Two got stuck at the same label.”

    Research turns opinions into better questions, better priorities, and better design decisions.

    Universal usability

    There is no single “average user.”

    1

    People vary

    Age, experience, goals, language, confidence, attention, and abilities all affect use.

    2

    Devices vary

    Phones, laptops, tablets, screen readers, slow networks, and different screen sizes change the experience.

    3

    Situations vary

    Users may be rushed, distracted, tired, outdoors, commuting, or trying to finish a task quickly.

    Discuss

    Which user situation is easiest for a designer to forget?

    Try this

    Same website. Different user context.

    User context

    A first-time visitor is using a phone in a parking lot.

    They need the address, hours, and whether walk-ins are accepted.

    NeedWhat is this user trying to accomplish?
    BarrierWhat could make the task harder?
    Design choiceWhat should be easier to see or do?
    TestHow could this be checked quickly?

    Research-directed process

    Hear → Create → Deliver

    H

    Hear

    Collect user stories, observe behavior, study goals, and listen before proposing solutions.

    C

    Create

    Group patterns, generate ideas, prioritize needs, and turn research into design directions.

    D

    Deliver

    Prototype, test, refine, and translate the strongest ideas into production plans.

    The first move is not “open the design tool.” The first move is listening.

    Activity

    Brainstorm first. Filter second.

    Prompt: A tutoring center wants more parents to request a consultation. Generate ideas before judging them.

    0 ideas selected
    Select ideas that could be explored as prototypes.

    Choosing ideas

    Good ideas still need filters.

    D

    Desirable

    Do users actually want or need this?

    F

    Feasible

    Can the team create it successfully with the available time and skills?

    V

    Viable

    Does it make sense for the client, resources, timeline, and project support?

    Research is not just about collecting ideas. It is also about deciding which ideas deserve to become prototypes.

    User research methods

    Different methods reveal different kinds of truth.

    Research caution

    What people say and what people do are not always the same.

    Surveys and interviews are useful, but observation can reveal behavior users forget to mention or cannot explain clearly.

    What a user might say

    “I always compare all the options carefully.”

    What observation might show

    The user scans the first three options, misses the filter, and chooses based on price and delivery time.

    Goal analysis

    Design for the goal, not just the task.

    Task thinking

    “The user needs to fill out a form.”

    Goal thinking

    “The user needs to request help quickly and feel confident someone will respond.”

    Discuss

    For a travel site, is the real goal “use the search form” or “find a trip that fits time and budget”?

    Build together

    Create a quick research-informed persona.

    A persona is a fictional representative user built from research, not guesswork. It keeps goals, motivations, barriers, and context visible while the team makes design decisions.

    Research-based stand-in

    Personas grow out of surveys, interviews, focus groups, and observation so the class can design for patterns in real users.

    Beyond the “average user”

    For universal usability, a strong persona should reflect differences in age, expertise, devices, access technology, and motivation.

    Persona preview

    Start filling in the persona to build a preview.

    Scenarios

    A scenario tells the story of a user trying to reach a goal.

    Scenario starter

    A parent visits a tutoring center website after work.

    They need to know if help is available for 9th grade algebra and how to request a consultation.

    StartWhere does the user begin?
    PathWhat steps would they try?
    FrictionWhere could they hesitate?
    SuccessHow do they know they completed the goal?

    Wireframes & prototypes

    Test the flow before polishing the visuals.

    Wireframe

    • Low-cost page structure
    • Navigation and content placement
    • Easy to change
    • Good for testing labels and flow

    Prototype

    • Connected screens or paper cards
    • Simulates a task
    • Lets users try the experience
    • Reveals confusion before code

    The cheaper a design is to change, the easier it is to learn from mistakes.

    Takeaway

    Research keeps design connected to real use.

    The goal is not to guess what users need. The goal is to listen, observe, prototype, test, and improve.
