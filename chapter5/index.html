    Chapter 5 · Site Structure

    The structure behind the screen

    A website is not only what visitors see. It is also the HTML, files, folders, content systems, links, metadata, and search structure that help the site work over time.

    Opening question What makes a website “well built” even before it looks good?

    Today’s path

    1. Separate content, presentation, and behavior
    2. Use semantic HTML to communicate meaning
    3. Structure documents with useful head and body content
    4. Organize files so the site can grow
    5. Understand CMS workflow and publishing roles
    6. Make pages findable through clean links, metadata, and SEO basics

    Use the arrow keys or the Next button to move through the deck.

    Main idea

    Site structure is the foundation under the design.

    Color, typography, and layout sit on top of a deeper structure. If the structure is weak, the site becomes harder to update, harder to search, harder to access, and harder to trust.

    H
    HTML

    Gives content meaning and structure.

    C
    CSS

    Controls presentation across pages and devices.

    J
    JavaScript

    Adds interactive behavior when needed.

    S
    Search

    Uses links, text, markup, metadata, and reliability to find pages.

    A strong site is not only attractive. It is understandable to browsers, screen readers, search engines, editors, and future developers.

    Separate the layers

    Content, presentation, and behavior should not be tangled together.

    A page is easier to maintain when each layer has a clear job. HTML describes the content. CSS styles it. JavaScript adds behavior.

    When the layers are separated, one change can improve many pages instead of becoming a one-page patch.

    .html
    Content and meaning

    Headings, paragraphs, lists, links, images, forms, sections.

    .css
    Presentation

    Layout, color, typography, spacing, responsive rules.

    .js
    Behavior

    Menus, interactions, validation, filtering, dynamic updates.

    assets
    Media and support files

    Images, icons, video, documents, fonts, downloadable files.

    Semantic markup

    HTML should describe what content means, not just how it looks.

    A heading marked as a heading, a list marked as a list, and a button marked as a button can be understood by more than the human eye.

    Weak structure

    <div class="big-text">Event Schedule</div>
<div>6:00 PM - Doors Open</div>
<div>7:00 PM - Music Starts</div>
<div onclick="go()">RSVP</div>

    Stronger structure

    <h1>Event Schedule</h1>
<ul>
  <li><time>6:00 PM</time> Doors Open</li>
  <li><time>7:00 PM</time> Music Starts</li>
</ul>
<button type="button">RSVP</button>

    Semantic markup supports accessibility, responsive display, search visibility, and cleaner maintenance.

    Head and body

    Every page needs structure that supports both humans and machines.

    The <head> prepares the page: title, metadata, stylesheets, scripts, and responsive settings. The <body> holds the content people interact with.

    A page title, headings, landmarks, links, image descriptions, and clean structure make the site easier to use and easier to find.

    Basic page skeleton

    <!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Vendor Lineup · Night Market</title>
  <meta name="description" content="Food, art, music, and student creators.">
  <link rel="stylesheet" href="assets/css/styles.css">
</head>
<body>
  <header>...</header>
  <main>...</main>
  <footer>...</footer>
</body>
</html>

    One shared stylesheet can adapt the same content to different screens.

    Responsive CSS changes layout without changing the meaning of the content.

    Files and folders

    A clear file structure makes the site easier to build, move, update, and debug.

    Files should live in predictable places. Names should be lowercase, descriptive, and stable. Important pages should have clear, readable URLs.

    A good folder system is boring in the best way: anyone on the team can understand where things belong.

    site/
    ├── index.html
    ├── about.html
    ├── schedule.html
    ├── vendors/
    │ ├── index.html
    │ └── taco-pop-up.html
    └── assets/
        ├── css/styles.css
        ├── js/app.js
        └── images/hero-market.jpg

    Content management systems

    A CMS is not only a page builder. It is a publishing system.

    A content management system can separate content from layout, manage roles, store reusable pieces, schedule updates, and create a workflow for writing, reviewing, approving, and publishing.

    RoleWriter

    Creates or edits content drafts.

    RoleReviewer

    Checks accuracy, tone, details, and completeness.

    RoleEditor

    Improves clarity, structure, and consistency.

    RolePublisher

    Approves release, scheduling, removal, and major updates.

    The more people involved, the more the website needs structure behind the scenes.

    Search visibility

    SEO starts with good structure, not tricks.

    Search engines read page text, links, titles, markup, metadata, URLs, image descriptions, and site maps. A messy site can hide good content from search.

    Good SEO overlaps with good accessibility and good organization: clear headings, descriptive links, useful content, alt text, readable URLs, and working pages.

    Helps findability

    • Clear page title
    • Descriptive meta description
    • Meaningful headings
    • Readable URLs
    • Linked pages and site map
    • Alt text for meaningful images

    Hurts findability

    • Broken links
    • Important text only inside images
    • Generic titles
    • Unclear link text
    • Slow or unreliable pages
    • Duplicate or outdated content

    Apply it

    Before a site is “finished,” check the invisible structure.

    1

    Can the content be understood without the styling?

    Headings, lists, links, and page sections should still make sense.

    2

    Can the files be understood by someone new?

    Folders, names, paths, and assets should be predictable.

    3

    Can the site be found, updated, and maintained?

    Search, metadata, links, CMS workflow, and reliability matter after launch.
